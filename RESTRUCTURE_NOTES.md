# Restructure Notes — versioned frontend/api layout

This document summarises the structural refactor performed on the
`restructure/versioned-frontend-api` branch.  It is intended for the repo
owner to review before merging, and flags every decision that was uncertain
or that diverged from the task brief.

---

## What moved and why

Goal: give frontend and API independent, versioned build units so that a
future `v2` (FastAPI + React/Vite) can be added as a clean sibling without
disturbing v1.

| Old path | New path | Notes |
|---|---|---|
| `index.php` | `frontend/v1/index.php` | |
| `submit.php` | `frontend/v1/submit.php` | |
| `admin/` | `frontend/v1/admin/` | all files, subdirs preserved |
| `config.php` | `frontend/v1/config.php` **and** `api/v1/config.php` | duplicated — see below |
| `Dockerfile` | deleted | replaced by `frontend/v1/Dockerfile` and `api/v1/Dockerfile` |
| `api/v1/*.php` | unchanged | already in the right place |
| `api/docs.php`, `api/index.php`, `api/acknowledgements.php` | unchanged | see API build-context note below |

Everything else (tests/, composer.json, phpunit.xml, playwright.config.js,
package.json, db/, docker/, .github/) stayed at the repo root.

---

## config.php — identical vs. diverged

**Decision: duplicated as-is (identical).**

Both `frontend/v1/config.php` and `api/v1/config.php` contain the same five
lines: they read `SITE_URL`, `MYSQL_HOST`, `MYSQL_USER`, `MYSQL_PASSWORD`,
`MYSQL_DATABASE` from env vars and expose them as PHP variables.  The
frontend and API consume the same variable names today, so the files are
identical.

A `shared/` directory was deliberately avoided: it would imply an ongoing
cross-version pattern, and a future `v2` should carry its own bootstrap,
not inherit v1 plumbing.  Each copy diverges as needed from this point.

---

## composer.json — dev-only, stays at repo root

`composer.json` was inspected: every package in it is a **dev dependency**
used only for testing and tooling:

- `guzzlehttp/guzzle` — HTTP client for PHPUnit integration tests
- `phpunit/phpunit` — test runner
- `symfony/finder`, `symfony/yaml` — transitive deps of swagger-php
- `zircote/swagger-php` — generates `api/v1/openapi.json` via composer script

None of these packages are required by the application at runtime.  The
production Docker images (`frontend/v1/Dockerfile`, `api/v1/Dockerfile`) do
**not** run `composer install`; no `vendor/` directory is baked in.

The `composer.json` / `composer.lock` pair stays at the repo root.  It is
used by CI (both PHPUnit and Playwright jobs install deps before running
tests) and by a developer running `vendor/bin/phpunit` or regenerating the
OpenAPI spec locally.

If the repo owner wants to remove these vestiges from the root (e.g. move
the test deps into a separate `tests/` or `tools/` composer project), that
is worth a dedicated cleanup pass — but it was not done here since the task
said to flag rather than remove.

## API build-context decision

The task brief specified `build context: api/v1` for the API service.
That build context was **expanded to `./api`** for the following reason:

Three files live at `api/` (not `api/v1/`):

- `api/docs.php` — Swagger UI page; tested by the Playwright suite
- `api/index.php` — API documentation page; tested by the PHPUnit suite
- `api/acknowledgements.php` — acknowledgements page; tested by Playwright

These files are conceptually part of the API service.  With a build context
of `api/v1/` only, Docker's `COPY` cannot reach them, so they would be
absent from the API container.

Using build context `./api` and dockerfile `v1/Dockerfile` captures all
three files.  The Dockerfile copies the entire build context to
`/var/www/html/api/`, preserving the `/api/` URL prefix inside the
container (matching the production Traefik path-routing model).

`docker-compose.yml` reflects this:
```yaml
api:
  build:
    context: ./api
    dockerfile: v1/Dockerfile
```

## Port / routing decision for local Docker dev

**Decision: two distinct ports — frontend on 8080, API on 8081.**

Rationale:
- Simpler `docker-compose.yml` — no routing layer (nginx/Caddy/Apache
  mod_proxy) required locally.
- The CI test suite uses a PHP built-in dev server with a custom router
  script (`tests/fixtures/router.php`) rather than the Docker stack, so
  local-dev port layout does not affect CI results.
- The router script was updated to serve `/api/...` from `./api/` and all
  other paths from `./frontend/v1/`, keeping all test URLs unchanged.

Trade-off accepted:
- Single-port (`http://localhost:8080/api/...`) more closely mirrors the
  production Traefik path-routing model.
- With two ports, API self-links in response bodies (generated by
  `api_base_url()` using `SITE_URL + '/api/v1'`) point to port 8081, which
  is where the API actually lives, so they resolve correctly.
- A future task adding Traefik labels / CD will naturally consolidate to a
  single public URL regardless of local-dev port layout.

Local dev URLs after this change:
- Frontend: `http://localhost:8080`
- API docs: `http://localhost:8081/api/`
- API example: `http://localhost:8081/api/v1/satellites.php`
- MariaDB: `localhost:3307`

## path reference fix in api/v1/lib/bootstrap.php

The only code change (other than config copies and new Dockerfiles) was a
single-line update to `api/v1/lib/bootstrap.php`:

```php
// Before:
require_once __DIR__ . '/../../../config.php';   // resolved to repo root

// After:
require_once __DIR__ . '/../config.php';          // resolves to api/v1/
```

`__DIR__` for that file is `api/v1/lib/`, so `../` goes to `api/v1/` where
`config.php` now lives.  The same relative path works correctly inside the
Docker container (where the build context `api/` is copied to
`/var/www/html/api/`).

- No Traefik labels, no CD workflow, no deployment.

`main`.
